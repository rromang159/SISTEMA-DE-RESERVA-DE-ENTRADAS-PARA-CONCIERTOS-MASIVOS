INTRODUCCIÓN
Actualmente, los sistemas informáticos cumplen un papel fundamental en la
organización de eventos masivos, especialmente en la venta y reserva de entradas para conciertos. Gracias a estas plataformas, los usuarios pueden adquirir boletos desde cualquier lugar de forma rápida y segura, evitando largas filas y mejorando la experiencia de compra.
Sin embargo, los conciertos de alta demanda presentan desafíos importantes para los
sistemas tecnológicos, ya que miles de usuarios pueden intentar acceder simultáneamente. Esto puede provocar errores, caídas del sistema o incluso la venta duplicada de entradas si no existe un adecuado control de concurrencia.
El presente trabajo tiene como finalidad analizar y diseñar una solución informática para
un sistema de reserva de entradas para conciertos masivos. Se abordarán aspectos esenciales como el manejo de usuarios concurrentes, seguridad, persistencia de datos, integridad de la información y procesamiento de pagos. Además, se elaborarán diagramas UML y se justificará técnicamente cada decisión tomada en el diseño del sistema.
METODOLOGÍA APLICADA-
Metodología de análisis y diseño
Para desarrollar este proyecto se utilizó una metodología de análisis y diseño orientada a objetos. Lo primero que se hizo es realizar un análisis del problema para identificar las necesidades relacionadas con la reserva de entradas para conciertos masivos. Luego, se realizó el levantamiento de requerimientos funcionales y no funcionales, se identificaron los actores y los procesos principales que se relacionan al sistema. Una vez de haber definido los requerimientos, se elaboraron diagramas UML para representar la estructura y el comportamiento del sistema. Se realizó el diagrama de casos de uso para identificar las funcionalidades requeridas y el diagrama de clases para modelar las entidades principales, sus atributos, métodos y relaciones. Finalmente, se aplicaron principios de diseño de software como abstracción, encapsulamiento, cohesión y bajo acoplamiento para proponer una solución escalable, segura y mantenible, justificando técnicamente cada una de las decisiones adoptadas.

ANÁLISIS DEL PROBLEMA. -
El principal problema en los sistemas de venta de entradas para conciertos masivos
ocurre cuando miles de personas intentan comprar boletos al mismo tiempo. Esta situación puede generar sobrecarga en el sistema, pérdida de información, errores de pago y conflictos relacionados con la disponibilidad de entradas.
Uno de los riesgos más importantes es la doble venta de boletos, la cual ocurre cuando
dos usuarios intentan comprar las mismas entradas simultáneamente. Para evitar este problema, el sistema debe implementar mecanismos de bloqueo temporal y control de concurrencia.
Además, es necesario garantizar:
Disponibilidad continua del sistema.
Actualización en tiempo real del inventario.
Seguridad de la información personal y financiera.
Integridad de los datos almacenados.
Respuesta rápida ante miles de solicitudes.
Por lo tanto, el sistema debe ser escalable, seguro y confiable para soportar eventos de
alta demanda sin afectar la experiencia del usuario.
Objetivo general. -
Diseñar e implementar una solución para un sistema de reservas de entradas para conciertos masivos a través de principios de arquitectura y diseño de software, con el único objetivo de garantizar seguridad, y evitar concurrencia en las ventas y así obtener una buena gestión de reservas.
Objetivos específicos. -

Diseñar diagramas UML.
Implementar fundamentos de arquitectura y diseño de software.
Garantizar la seguridad e integridad de los datos y del sistema.

PRINCIPIOS DE DISEÑO APLICADOS
Abstracción. - Muestra las características más importantes de un objeto
o sistema, ocultando los detalles innecesarios o complejos.
En un sistema de reserva de entradas para conciertos masivos, la abstracción permite que el usuario únicamente visualice funciones simples como seleccionar conciertos, escoger asientos y realizar pagos, mientras el sistema oculta procesos internos complejos. La plataforma realiza validaciones de disponibilidad, control de reservas, procesamiento de pagos, autenticación de usuarios y actualización automática de los asientos. Es decir, cuando el usuario presiona “Reservar asiento”, el sistema ejecuta automáticamente varios procesos sin mostrar su funcionamiento interno. Gracias a la abstracción, el sistema reduce su complejidad, mejora la organización del software y facilita tanto el desarrollo como la experiencia del usuario.
Encapsulamiento. - Consiste en agrupar dentro de una misma clase los
datos o atributos y los métodos que trabajan sobre ellos. Además, permite controlar el acceso a la información interna de un objeto, evitando que otros elementos del sistema modifiquen directamente sus datos.
Protege información importante como reservas, pagos y disponibilidad de asientos. Por ejemplo, el estado de un asiento no puede modificarse directamente, sino únicamente mediante métodos controlados como reservarAsiento(), los cuales validan la disponibilidad antes de realizar la reserva. Además, los datos de usuarios y pagos se mantienen privados para evitar accesos no autorizados y reducir errores. Gracias al encapsulamiento, el sistema mejora la seguridad, evita reservas duplicadas y facilita el mantenimiento del software.
Descomposición. - Consiste en dividir un sistema complejo en partes más
pequeñas, organizadas y manejables, llamadas módulos o componentes, cumpliendo una función específica, es decir; se descompone por Módulo de usuarios, Módulo de conciertos, Módulo de entradas, Módulo de reservas, Módulo de pagos, Módulo de confirmación y ventas, Módulo administrativo, Módulo de base de datos.
Cohesión. - Si la cohesión es alta, significa que el módulo es más fácil de
entender, mantener y modificar, porque sus responsabilidades están bien definidas, En el sistema la cohesión se evidencia en que cada módulo ya que tienen responsabilidades bien definidas.
Acoplamiento. - Evalúa el grado de interdependencia, conectividad y
dependencia mutua existente entre dos o más módulos, clases o componentes. Es decir, Permite que módulos como usuarios, pagos, reservas y conciertos funcionen de manera independiente. Por ejemplo, el módulo de reservas únicamente envía los datos necesarios al módulo de pagos para procesar la compra, sin depender directamente de su funcionamiento interno. Gracias a esto, el módulo de pagos puede actualizarse o modificarse sin afectar al módulo de reservas ni al resto del sistema. El bajo acoplamiento mejora el mantenimiento, reduce errores y facilita que varios desarrolladores trabajen simultáneamente en diferentes módulos del sistema.
DISEÑO DE SOLUCIÓN
1. Diagrama de Clases UML
Usuario: El cliente que compra.
Concierto: El evento que tiene entradas.

más lo tome mientras pagas.
Pago: La transacción de dinero.
2. Persistencia de Datos (Base de Datos)
Es donde se guarda la información de forma segura en MySQL o PostgreSQL. Lo clave
aquí son las propiedades ACID (Transacciones seguras). Sirven para que si dos personas dan clic al mismo asiento al mismo tiempo, el sistema congele la base de datos por un milisegundo, se lo dé al primero y al segundo le diga que ya no está disponible, evitando la doble venta. fig.5
3. Seguridad básica
Cifrado: Las contraseñas se guardan ocultas con BCrypt (nunca en texto
plano).
Roles: El Cliente solo puede comprar; el Administrador puede crear
conciertos y cambiar precios.
Validaciones: El sistema verifica en el servidor que nadie altere el precio
de la entrada desde el navegador web antes de pagar.
Manejo de errores
Usa bloques try-catch para que la página no se caiga ni se quede en blanco cuando ocurre un imprevisto:
Error de Clave: Dice “Credenciales incorrectas” sin avisar si el correo
existe, por seguridad.
Error de Concurrencia: Si te ganan el boleto, avisa: “Asiento no
disponible”.
Error de Tiempo: Si tardas más de 10 minutos en pagar, cancela la
reserva y libera los boletos para otros compradores.
JUSTIFICACIÓN TÉCNICA
La arquitectura del sistema de reserva de entradas para conciertos masivos se fundamenta en un diseño robusto y estructurado, diseñado para garantizar la operatividad bajo condiciones de alta demanda. La elección de los componentes técnicos se justifica bajo los siguientes criterios:
En primer lugar, se utiliza el estándar UML (Unified Modeling Language) como herramienta de modelado esencial. Su implementación permite visualizar y documentar la estructura del sistema antes de su construcción, asegurando una comunicación clara entre los actores del proyecto y permitiendo identificar fallos de lógica de manera temprana, lo que garantiza que el flujo de compra sea coherente y eficiente.
Para soportar la carga transaccional, el sistema se ha dividido en módulos independientes. Esta modularización facilita la mantenibilidad, permitiendo realizar actualizaciones en áreas específicas (como la pasarela de pagos) sin comprometer el resto de la plataforma. Además, otorga escalabilidad al sistema, permitiendo que los módulos críticos, como el de reservas, reciban más recursos durante los picos de tráfico sin degradar el rendimiento general.
Módulo usuarios
Gestiona el registro e inicio de sesión y controles de perfiles de los usuarios.
Módulo conciertos
Gestiona toda la información relevante de los conciertos.
Módulo reservas
Gestiona la selección de bloqueo temporal, confirmación de entradas y así evitar la venta repetida de asientos.
Módulo pagos
Gestiona las compras y valida los pagos.
La seguridad se establece como un pilar transversal e innegociable. Mediante la
implementación de protocolos HTTPS, cifrado de datos y validaciones de identidad, el sistema protege la integridad de la información sensible de los usuarios y previene fraudes electrónicos. Esto no solo cumple con las normativas legales de protección de datos, sino que construye un entorno de confianza para el consumidor final.
Finalmente, el uso de una base de datos relacional es la pieza clave para la consistencia del inventario. Gracias a las propiedades ACID (Atomicidad, Consistencia, Aislamiento y Durabilidad), el sistema garantiza que cada entrada sea única y que no existan errores de doble venta. La base de datos permite gestionar con precisión milimétrica la relación entre usuarios, asientos y eventos, asegurando que cada transacción se registre de forma persistente y segura, incluso ante fallos inesperados del servidor.

CONCLUSIONES
La ejecución de este informe técnico permite concluir que el éxito de un sistema de reserva de entradas para eventos masivos depende de una arquitectura que priorice la consistencia y la seguridad de los datos. En primera instancia, se determina que el análisis de requerimientos fue el paso crítico para comprender que el mayor desafío no es la venta en sí, sino el manejo de la altísima concurrencia, lo que permitió diseñar estrategias específicas para evitar la degradación del servicio bajo presión.
Desde la perspectiva de los principios de diseño, se concluye que la organización modular del sistema es lo que garantiza su estabilidad. Al separar las responsabilidades, se logra que procesos complejos, como el bloqueo temporal de asientos, funcionen con precisión quirúrgica, asegurando que el sistema sea escalable y fácil de mantener a largo plazo. Esta estructura previene errores en cadena y permite actualizaciones sin interrumpir la operación global.
Sobre la implementación de UML, se concluye que su uso fue fundamental para transformar las necesidades de negocio en un plano técnico ejecutable. El modelado visual permitió anticipar la interacción entre los usuarios, el administrador y la pasarela de pagos, eliminando ambigüedades y asegurando que cada flujo crítico fuera validado antes de la fase de construcción.
Finalmente, la solución propuesta basada en transacciones seguras y el control estricto de inventario en tiempo real se consolida como la respuesta definitiva al problema de la "doble venta". En conclusión, el sistema diseñado no solo cumple con las expectativas funcionales, sino que ofrece una plataforma robusta, segura y confiable que garantiza la integridad de cada transacción y una experiencia de usuario óptima.
